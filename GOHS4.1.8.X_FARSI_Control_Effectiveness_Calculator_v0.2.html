1. Purpose

Overview

This workbook calculates the residual likelihood reduction achieved when controls are applied to a defined hazard. It supports the Inherent / Current staged approach endorsed by ISO 45002:2023 Cl 6.1.2.2 guidance. It is NOT a prescribed ISO 45001 / ISO 45002 method. Output is transferred to the parent risk assessment in Figtree or the GOHS4.1.8 risk worksheet.

Goal

To quantify how much each control reduces inherent risk, producing a defensible, auditable residual risk score.

Compliance

Tool draws on barrier-management performance assessment principles (FARSI) and applies hierarchy-of-controls protection factors consistent with ISO 45002:2023 Cl 6.1.2.2 guidance. SFARP demonstration, hazard identification, consultation evidence, additional controls, and target risk are recorded in the parent risk assessment, not in this tool.

2. Assessment Methods

FARSI Method (Quantitative – Detailed)

What it isScores 5 independent attributes: F–Functionality, A–Availability, R–Reliability, S–Survivability, I–Interaction. Each scored 1–5 on evidence.
How it worksAverage of 5 scores → Effectiveness = (Score−1) ÷ 4. Score 5 = 100%; Score 1 = 0%.
Evidence RequiredFunctional testing, maintenance logs, historical performance data, stress testing, system integration testing. See FARSI Scoring Criteria tab.
Key AdvantageIdentifies specific weaknesses. E.g., F=5, A=5, R=1, S=5, I=5 reveals a critical Reliability gap that Simple would miss.

Simple Method (Qualitative – Quick)

What it isSingle 1–5 rating based on overall professional judgement.
How it works5=Highly Effective, 4=Effective, 3=Moderately Effective, 2=Partially Effective, 1=Not Effective. Same formula: (Score−1)/4.
Evidence RequiredGeneral assessment based on observation, documentation review, or expert opinion.
Key LimitationMay miss hidden weaknesses. Assessor may rate 4–5, overlooking specific attribute gaps.

5. Sequential Reduction Principle

PrincipleEach control reduces risk from the residual level left by the prior control, NOT from the inherent level. This reflects real-world stacking of controls.
ExampleInherent L=4 → Control 1 (Eng, 75% eff) → Residual 2.2 → Control 2 (Admin, 50% eff) → Residual 1.76 → Final Risk = 1.76 × Consequence
Consequence StabilityConsequence remains stable unless a control physically limits the harm mechanism. Most controls reduce Likelihood only.
Control OrderApply higher-hierarchy controls first (Elimination → Substitution → Engineering → Admin → PPE) for maximum cumulative reduction.

6. Calculation Formulas

Effectiveness MultiplierBanded by score  |  ≥4.0=100% (High), 3.0–3.9=60% (Moderate), 2.0–2.9=30% (Low), <2.0=10% (Very Low)
Reduction Potential1 − Hierarchy Base Factor  |  Elimination 100%, Substitution 80%, Engineering 60%, Admin 40%, PPE 20%
Effective ResidualBase Factor + (Reduction Potential × (1 − Effectiveness Multiplier))
Independence CombinationIf not independent of the prior control: use MIN(prior residual, this residual) once, applied to the likelihood that fed the prior control — does not compound
Current LikelihoodPrior Likelihood × Effective Residual (or combined residual, see above)
Residual Likelihood FloorCurrent Likelihood cannot fall below Inherent Likelihood × 20% (general) or × 10–15% (ADI / high-consequence) across any non-Elimination control chain
Current RiskCurrent Likelihood × Inherent Consequence (fractional – internal use only)
Rounded Current RiskRound(Current Likelihood, nearest integer, floor 1) × Consequence
% Reduction(Inherent Risk − Current Risk) ÷ Inherent Risk × 100

Effectiveness banding, Residual Likelihood Floor, and Independence Combination implement the FARSI Reduction Factors model — see Mandatory Model Safeguards on the Reference Tables tab.

4. How to Use This Workbook — Step by Step

  1. Define Hazard: Go to the Control Reduction Calculator tab. Enter hazard/risk description. Select Inherent Likelihood (1–5) and Consequence (1–5). Inherent Score and Risk Rating calculate automatically.
  2. Add Control ID: Click "+ Add Control Row". Enter a unique Control ID (e.g., CTRL-001, RC2342).
  3. Select Control Type: Choose from dropdown — Elimination (0.0), Substitution (0.2), Engineering (0.4), Admin (0.6), PPE (0.8). This sets the base residual factor.
  4. Select Method: Choose "FARSI" for detailed or "Simple" for quick rating.
  5. If FARSI: Score each attribute F, A, R, S, I from 1–5 using the FARSI Scoring Criteria tab. Score on evidence, not assumption.
  6. If Simple: Enter single rating 1–5 in the Simple Rating field.
  7. Review Results: Check Effectiveness %, Effective Residual, Current Likelihood, Current Risk, Risk Rating, and % Reduction — all calculated automatically.
  8. Add More Controls: Repeat Steps 2–7. Each control reduces risk sequentially from the prior level.
  9. Review Summary: Check the Summary section for Inherent vs Current Risk, including ROUNDED integer outputs for Figtree transfer.
  10. Complete Traceability: Fill in Calculation ID, Parent RA ID, Figtree Hazard ID, Assessor, Date, Reviewer before transferring outputs.

8. Best Practices & Warnings

✅ DO

  • Always score on documented evidence, not assumptions. If evidence unavailable, score conservatively and note the gap.
  • Apply higher-hierarchy controls first for maximum cumulative reduction.
  • Watch for attribute variance — one low score (e.g., R=1) reveals a specific weakness.
  • Document rationale for each score, especially for critical risks.
  • Use a separate calculation for each hazard.
  • Re-assess controls after incidents, near-misses, or process changes.

⚠️ WARNING

  • Do NOT assume effectiveness because a control exists — verify through inspection, testing, or performance data.
  • Do NOT mix FARSI and Simple for a single control.
  • Do NOT transfer raw fractional Current Likelihood to Figtree — use the ROUNDED integer.
  • Do NOT cite this calculation as SFARP — SFARP belongs in the parent RA.
  • Where controls share a failure mode, record in the Independence Check field and treat as composite.

Methodology Notes

Status of formula: Effectiveness = (Score−1)/4 and multiplicative residual formulas are calculation aids. NOT prescribed by ISO 45001:2018, ISO 45002:2023, or ISO 31000:2018. ISO 45002:2023 Cl 6.1.2.2 endorses the Inherent / Current / Target staged approach.

Assumption 1 – Independence: Sequential multiplicative reduction assumes controls fail independently. Where two controls share a failure mode, record in Independence Check and treat as a composite control.

Assumption 2 – Hierarchy Compounding: Validate against historical incident data before using calculated Current Risk as basis for a band change.

Assumption 3 – Consequence Stability: Consequence remains stable unless a control physically limits the harm mechanism (e.g., fall arrest reduces fatal fall to broken bone).

Assumption 4 – Continuous Likelihood: Calculation produces fractional Current Likelihood (e.g., 1.76). The GOHS4.1.8 risk matrix uses integer likelihood (1–5). Round to nearest integer — floor at 1 (never report 0).

Limitations: This tool does not assess hazards, identify causes, document consequence justifications, prescribe additional controls, set target risk, or generate the SFARP statement. These belong in the parent risk assessment.

Mandatory Model Safeguards

1. Residual Likelihood FloorNo single non-Elimination control (or combination) may reduce likelihood below 20% of inherent likelihood (general hazards) or 10–15% of inherent likelihood (ADI / high-consequence hazards). Prevents one control declaring residual risk "acceptable" on its own. Set the hazard's ADI flag and floor % in Step 1 — Hazard & Risk Definition.
2. Sequential Application OnlyApply controls top-down: Elimination → Substitution → Engineering → Admin → PPE. Each reduction acts on the remaining residual likelihood, not the inherent likelihood. The calculator shows a warning banner (non-blocking) if rows are entered out of order.
3. Independence CheckIf a control is not independent of the prior control (shares a common-cause failure mode), they are treated as one combined control: the calculator uses the lower (more protective) residual once, instead of multiplying both — avoiding artificial stacking credit. Uncheck "Independent" on the control row and document the rationale in the note field.

Overview — The FARSI Framework

The FARSI control effectiveness process is a structured framework used in safety risk management to define, measure, and verify the performance of safety-critical controls. It ensures that controls are not only present but effective, reliable, and auditable throughout their lifecycle. It transforms vague risk reduction into structured, measurable, and verifiable performance standards.

AttributeLetterCore QuestionVerification Methods
FunctionalityFDoes the control perform its intended safety-critical function under expected conditions?Functional testing, design documentation, simulation/scenario walkthroughs
AvailabilityAIs the control accessible and operational when needed, especially during emergencies?Maintenance logs, inspection schedules, real-time system status checks
ReliabilityRDoes the control consistently perform its function over time and across conditions?Historical performance data, reliability engineering analysis, incident reports
SurvivabilitySCan the control remain functional during and after adverse conditions (fire, explosion, flood)?Stress testing, environmental qualification, fire and blast rating assessments
InteractionIDoes the control integrate effectively with other systems, controls, and human operators?System integration testing, HMI reviews, cross-control dependency mapping

F — Functionality

Definition: The control must perform its intended safety-critical function under expected conditions.

Key Questions: Does the control meet its design intent? Is it capable of preventing or mitigating the hazard? Are failure modes understood and accounted for?

Example: A gas detector must reliably trigger alarms when thresholds are exceeded — not just exist in the system.

ScoreLevelCriteria / Evidence Indicators
1Poor / Non-compliantControl does not perform intended safety function or is misconfigured. No functional test records exist. Design intent cannot be verified.
2Below AdequateControl performs function only under ideal or limited conditions. Significant gaps between design intent and actual performance documented.
3Adequate / Partially CompliantControl performs function inconsistently or under limited conditions. Some functional test records available. Known gaps exist.
4GoodControl performs function consistently with minor gaps or conditions. Functional test records current. Minor design limitations identified and managed.
5Excellent / Fully CompliantControl reliably performs intended safety function under all expected conditions. Full functional certification, automated monitoring data, and predictive maintenance records current.

A — Availability

Definition: The control must be accessible and operational when needed, especially during critical phases or emergencies.

Key Questions: Is the control online and active? Are there any known outages or bypasses? Is it maintained and inspected regularly?

Example: A fire suppression system must be charged and ready — not under maintenance or disabled.

ScoreLevelCriteria / Evidence Indicators
1PoorControl is unavailable, bypassed, or frequently offline. No maintenance schedule. Multiple known outages without resolution.
2Below AdequateControl is often unavailable during peak risk periods. Maintenance schedule exists but compliance poor. Bypasses documented but not managed.
3AdequateControl is intermittently available or has limited access during critical phases. Maintenance schedule followed most of the time. Occasional planned outages managed.
4GoodControl is generally available with occasional scheduled downtime that is managed safely. Maintenance records current and compliant.
5ExcellentControl is consistently available and accessible when needed. Full maintenance compliance records, real-time status monitoring, bypass management procedure in place.

R — Reliability

Definition: The control must consistently perform its function over time and across conditions.

Key Questions: What is the failure rate or Mean Time Between Failures (MTBF)? Are there redundancies or backups? Has it failed in the past?

Example: A pressure relief valve that sticks intermittently is unreliable — even if it passed initial testing.

ScoreLevelCriteria / Evidence Indicators
1PoorControl fails frequently or lacks redundancy. MTBF low or unknown. Multiple documented failures with no root cause analysis or corrective action.
2Below AdequateControl has significant failure history with limited backup measures. MTBF below acceptable threshold. Some incident reports exist but corrective actions incomplete.
3AdequateControl has moderate failure rate or limited backup systems. MTBF acceptable. Incident reports exist with some corrective actions implemented.
4GoodControl has low failure rate with some redundancy in place. MTBF within acceptable range. Reliability engineering analysis available.
5ExcellentControl performs consistently with low failure rate and robust redundancy. High MTBF documented. Full reliability analysis, historical performance data, and backup systems verified.

S — Survivability

Definition: The control must remain functional during and after adverse conditions (e.g., fire, explosion, flood, seismic event).

Key Questions: Can it withstand environmental stressors? Is it protected from cascading failures? Does it have post-event functionality?

Example: A control panel located in a blast zone must be shielded or remotely operable to survive an incident.

ScoreLevelCriteria / Evidence Indicators
1PoorControl is vulnerable to environmental stressors (fire, blast, flood). No environmental qualification. Location or design exposes control to likely failure during the incident it is meant to mitigate.
2Below AdequateControl has minimal protection with known vulnerability to adverse events. Partial shielding or location mitigation only. Post-event functionality not verified.
3AdequateControl has partial protection or limited survivability. Environmental qualification partial. Some stress testing conducted.
4GoodControl is mostly protected with minor exposure risks under extreme events. Fire/blast ratings available. Post-event testing conducted periodically.
5ExcellentControl is fully protected and remains functional under adverse conditions. Full environmental qualification, fire and blast rating certification, remote operability, post-event functionality verified.

I — Interaction

Definition: The control must integrate effectively with other systems, controls, and human operators.

Key Questions: Does it communicate with upstream/downstream systems? Are there interface issues or delays? Is it intuitive for operators?

Example: An emergency shutdown system must coordinate with alarms, valves, and operator panels — not act in isolation.

ScoreLevelCriteria / Evidence Indicators
1PoorControl does not integrate with other systems or causes conflicts. System integration testing not conducted. Known interface failures or operator confusion documented.
2Below AdequateControl has significant interface issues or creates dependency conflicts. Partial integration only. HMI review reveals usability issues. Cross-control dependency not mapped.
3AdequateControl interacts with some systems but has interface issues. System integration testing partially complete. Some HMI issues identified and being addressed.
4GoodControl integrates well with most systems with minor coordination issues. System integration test records available. HMI review completed. Cross-control dependency mapped.
5ExcellentControl integrates seamlessly with all other systems and supports coordinated emergency response. Full system integration certification, HMI design verification, operator training verified, cross-control dependency mapping complete.
